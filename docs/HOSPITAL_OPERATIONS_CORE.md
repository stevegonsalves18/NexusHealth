# Hospital Operations Core

This module is the foundation for a full AI-assisted hospital and clinic operating system. It supports department workflows, OPD/IPD/emergency encounters, admissions, beds, clinical orders, patient timelines, doctor panels, and admin operations summaries.

## Product Boundary

The system organizes real-time and batch-operational healthcare data for patients, doctors, and administrators. It does not make autonomous medical decisions.

Safe positioning:

> AI-assisted hospital and clinic workflow intelligence with clinician-reviewed insights.

Do not claim:

- autonomous diagnosis
- autonomous emergency triage
- automated treatment decisions
- replacement of doctors, nurses, or hospital staff
- regulator-cleared medical-device behavior

## Core Concepts

| Concept | Purpose |
| --- | --- |
| Department | OPD, IPD, emergency, diagnostics, pharmacy, nursing, or administration unit |
| Bed | Ward/bed capacity and occupancy tracking |
| Encounter | OPD/IPD/emergency visit context for a patient and clinician |
| Admission | Inpatient stay tied to an encounter and optional bed |
| Clinical Order | Lab, radiology, pharmacy, nursing, procedure, or department task |
| Care Event | Timeline event generated by workflow activity |

## Role Views

Patient view:

- Own timeline only.
- Own encounters, admissions, orders, and care events.
- AI and operational insights remain informational and clinician-reviewed.

Doctor view:

- Assigned patient panel.
- Open encounters.
- Open department orders.
- Active admissions.
- Deterministic workflow insights and review prompts.

Admin view:

- Department and bed setup.
- Operations counts.
- Encounter/order/admission summaries.
- Audit and readiness views from adjacent modules.

## Implemented API Surface

Departments:

- `POST /hospital/departments` - admin only
- `GET /hospital/departments` - authenticated users

Beds:

- `POST /hospital/beds` - admin only

Encounters:

- `POST /hospital/encounters` - doctor/admin only

Admissions:

- `POST /hospital/admissions` - doctor/admin only

Orders:

- `POST /hospital/orders` - doctor/admin only

Views:

- `GET /hospital/patient/timeline` - current patient only
- `GET /hospital/doctor/patients` - assigned doctor panel, admin can inspect all
- `GET /hospital/doctor/insights` - workflow insights, not diagnosis
- `GET /hospital/admin/operations` - admin operations summary

## Data Engineering Direction

Real-time layer:

- Every operational action should emit a care event.
- Dashboards can subscribe to recent events.
- Doctors and administrators see live operational state.

Batch layer:

- Aggregate department utilization.
- Detect follow-up gaps.
- Track order completion delays.
- Find recurring population-level screening patterns.
- Produce clinic closeout and performance reports.

Clinical safety:

- Pattern language must say "observed", "flagged", "suggests review", or "needs clinician review".
- It must not say "diagnosed", "confirmed", or "treatment required" unless a clinician records that decision.
